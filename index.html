Mold Maintenance, Release & Part Pull Tracker

Power Apps Canvas · Mold Release · Rework Tracking · Part Pull Counters · Attachments

The problem

Molds moved through release, part-pull, and rework activities without a clear operational screen for tracking counters, release status, rework history, and supporting evidence. The risk was that technicians could keep pulling parts after a release window expired, or rework could happen without the mold record being updated consistently.

My contribution

  • Built a Power Apps canvas mold tracker with three focused experiences: new mold upload, mold maintenance/editing, and release/part-pull recording.
  • Designed the Mold Maintenance screen with search, status filters, selected mold details, editable information, attachments, and read-only tracking counters.
  • Added release and rework counters such as parts pulled since last release, parts pulled since latest rework, total pulls, total releases, and total reworks.
  • Created the Mold Release & Part Pull screen so floor users can scan or enter a mold number, review the selected mold, then record a release or part pull from one controlled page.
  • Implemented visual release warnings and reset behavior so when a release is applied, the parts-pulled-since-release counter returns to zero and technicians can see the mold is ready again.
  • Added a New Mold Upload form for registering MLD number, title/name, description, NCR reference, and attachments before release/rework tracking begins.

Stack

Outcome

A single operational workspace for mold release and rework control, giving technicians and planners visibility into whether a mold is OK, needs release, has active rework history, or requires action before additional parts are pulled.

Enterprise Purchase Request Approval System

Model-Driven · C# Plugin · Approver Matrix · Security · Power BI

The problem

Procurement approvals lived in email threads, spreadsheets, and manual reminders — no central status, threshold enforcement by memory, sensitive financial fields overexposed, and no audit trail or cycle-time visibility. The real gap was governance: approvals were treated as messages, not controlled business records.

My contribution

  • Designed the Dataverse data model — request header, line details, an approval-transaction table, and a configurable Approver Matrix (department · level · approver · threshold) with supporting lookups.
  • Wrote a C# plugin to enforce approval progression and status transitions server-side after every approver decision — reliable and transactional.
  • Built the model-driven UI with JavaScript + Ribbon Workbench command-bar actions (Submit / Approve / Reject / Populate), enable/display rules, and subgrid buttons.
  • Layered governance — security roles, access teams, field-level security on financial data, and audit history — plus Power BI KPIs for cycle time, throughput, spend, and SLA.
  • Packaged it for ALM: managed-solution promotion across dev → test → UAT → production.

06 — Dataverse Business Skills via MCP

I turn Dataverse tables into conversational business skills

Using the Dataverse MCP server, I expose manufacturing tables as natural-language skills — so a Copilot agent can answer operations questions and act on records without anyone opening the ERP.

How I add them — via Dataverse MCP

1

Entra app registration

Register an app with the Dynamics CRM → mcp.tools permission and a client secret; grant admin consent.

2

Enable Dataverse MCP

On a Managed Environment, enable Dataverse MCP, add an application user, and grant a security role scoped to the tables.

3

Register as a plugin

Wire an OAuth client and package the MCP endpoint as a Copilot plugin/connector via the OAuth vault.

4

Toggle the skill on

Upload to the agent surface (Copilot Studio or Copilot 365 Cowork) and enable the skill in Sources & Skills.

⚡

How they're triggered: the agent's orchestrator matches a user's natural-language intent to the skill's description and synonyms, then the Dataverse MCP runs a security-trimmed query over the table and returns structured rows — optionally summarized by an AI Builder Run-a-Prompt. Read-only by default and always governed by the caller's Dataverse security roles, so no one sees data they shouldn't.
